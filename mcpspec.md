# AI Hype Machine — Identity, Persistence & MCP Redesign Spec

## Problem Statement

Three linked issues:
1. **Ephemeral DB** — SQLite lives on Railway's disposable disk. Every redeploy wipes all player data.
2. **Fragile identity** — Player UUID lives only in `localStorage`. Clear browser data or switch device = lost account. No recovery path.
3. **Brittle MCP connection** — MCP has a hardcoded UUID in `.mcp.json`. Breaks on every DB wipe and is always a separate player from the browser, requiring manual sync.

---

## Goals

- Player data survives Railway redeploys indefinitely
- Players can recover their account on any device using name + PIN
- MCP connects to any player by name + PIN — no UUID config needed
- Browser and MCP are always the same player, switching seamlessly
- Users can create multiple accounts and connect MCP to any of them at will

---

## 1. Persistence — Migrate SQLite → Supabase

### Why Supabase

- Already have a spare Supabase project available
- Managed PostgreSQL — data persists independently of Railway
- Supports `jsonb` column for the existing state blob (no schema explosion)
- No change to game logic — only the DB adapter changes

### Schema

One table, mirrors the current SQLite structure with two additions (`pin`, `pin_hash`):

```sql
CREATE TABLE players (
  id          TEXT PRIMARY KEY,                         -- UUID, generated by server
  name        TEXT NOT NULL,                            -- display name, not unique
  pin_hash    TEXT NOT NULL,                            -- bcrypt hash of 4–8 digit PIN
  state       JSONB NOT NULL,                           -- full GameState blob
  score       BIGINT NOT NULL DEFAULT 0,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_players_score ON players (score DESC);
CREATE INDEX idx_players_name  ON players (lower(name));  -- case-insensitive lookup
```

### DB Adapter

- New file: `server/src/db/supabase.ts` — replaces `server/src/db/index.ts`
- Uses `@supabase/supabase-js` with service role key (server-side only)
- Env vars on Railway: `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`
- Interface stays identical to current DB module — drop-in replacement, no engine changes

### Save Strategy

Keep the current 2-second debounced save. Supabase Postgres can handle the write frequency of an idle clicker easily.

## 2. Identity — Name + PIN

### Design Principles

- Name is a display label, not a unique key. Multiple accounts can share the same name (e.g. "kamil" on different devices = different players if different PINs).
- UUID is the true identity. Name + PIN is the *recovery key* to look up that UUID.
- PIN is hashed (bcrypt, cost 10) — good enough for a casual game, not over-engineered.
- No email, no OAuth — keep it frictionless.

### Name + PIN → UUID Resolution

On any authentication attempt, the server does:

1. Look up players by `lower(name) = lower(input_name)` — may return multiple rows (names are not unique)
2. For each match, compare `bcrypt(input_pin, stored_hash)`
3. First match wins → return that player's UUID
4. No match → error: "Wrong name or PIN"

This means: if two players both named "kamil" exist, the first one whose PIN matches wins. Acceptable for a casual game; tell users to pick a distinctive name if they care.

### Registration Flow

**Endpoint:** `POST /api/players`
**Body:** `{ playerName, pin }`
**Behaviour:** Always creates a new player. Does not check for duplicate names.

```
Server:
1. Validate: name 1–20 chars, pin 4–8 digits only
2. Generate UUID
3. Hash PIN
4. Insert row → return { playerId, state }

Client:
5. Store { playerId, playerName } in localStorage
6. Show game
```

### Login / Recovery Flow

**Endpoint:** `POST /api/auth`
**Body:** `{ playerName, pin }`
**Behaviour:** Resolves existing player by name + PIN.

```
Server:
1. Look up by name (case-insensitive)
2. Compare PIN hashes
3. Return { playerId, state } on match, 401 on failure

Client:
4. Store { playerId, playerName } in localStorage
5. Continue game from existing state
```

### Browser Reconnect Flow (unchanged)

If `localStorage` has `ai_hype_player_id`:
- Connect WebSocket with stored UUID directly — fast path, no PIN needed
- Server loads session from Supabase if not in memory

If `localStorage` is empty (cleared, new device, new browser):
- Show a split UI: **Register** (new account) or **Log in** (existing account)
- Login form: name + PIN → call `POST /api/auth` → store UUID → enter game

### WebSocket Auth (minimal change)

Current WebSocket just accepts any `playerId` in query params — no verification.

Proposed: no change for now. Once the client has a UUID via login/register, it uses it directly. The UUID is not guessable in practice (UUID v4). Adding PIN verification to WS handshake is a follow-up hardening step, not needed for v1.

## 3. MCP Identity Redesign

### Current Problem

`.mcp.json` hardcodes `MCP_PLAYER_ID` — a UUID that breaks on every DB wipe and is always a different player from the browser.

### Proposed Config

```json
{
  "mcpServers": {
    "ai-hype-machine": {
      "command": "node",
      "args": ["server/dist/mcp/index.js"],
      "env": {
        "MCP_PLAYER_NAME": "kamil",
        "MCP_PLAYER_PIN": "1234",
        "API_URL": "https://www.tikkensgame.com/api"
      }
    }
  }
}
```

No UUID. Name + PIN. That's it.

### MCP Startup Behaviour

On `main()`, before connecting transport:

```
1. POST /api/auth { playerName, pin }
2. On success → store playerId in memory for the session
3. On 401 → log error "Wrong name or PIN" and exit
4. On network error → retry 3x with 2s backoff, then exit
```

The MCP never needs to know the UUID in advance. It resolves it on startup from the same API the browser uses. If the player was just created in the browser, the MCP will find them immediately.

### Workflow: Switching Players

To point MCP at a different player, user changes `MCP_PLAYER_NAME` + `MCP_PLAYER_PIN` in `.mcp.json` and restarts Claude Code from the game directory. No UUID hunting needed.

### MCP Tools — No Changes

All existing tools (`click`, `click_n`, `buy_producer`, etc.) remain unchanged. They use the in-memory `playerId` resolved at startup.

---

## 4. Co-pilot Mechanics (No Code Change Needed)

The "AI plays for you, you can take over" behaviour already works:
- Both browser and MCP hit the same `/api/*` endpoints for the same `playerId`
- State updates flow to the browser via WebSocket regardless of who triggered them
- `X-Mcp-Source: 1` header triggers the visual flash in the browser when MCP acts

No new code needed here. The identity fix (same player in both places) is what makes this feel seamless.

---

## 5. Out of Scope for This Change

- WebSocket PIN verification (low priority, UUID is not guessable)
- Rate limiting on `/api/auth` (nice-to-have, not blocking)
- Multiple named accounts per browser session (overkill for v1)
- Supabase Realtime (not needed — Railway WebSocket handles push)
- Any changes to game engine, tick loop, producers, upgrades, or prestige

---

## Implementation Order

1. **Supabase setup** — create table, get credentials, wire env vars on Railway
2. **DB adapter** — replace `db/index.ts` with Supabase client, same interface
3. **PIN hashing** — add `bcryptjs` dependency, update player create/auth
4. **New endpoint** — `POST /api/auth` (login/recovery)
5. **Update `/api/players`** — require PIN in body, hash before insert
6. **MCP startup** — replace hardcoded UUID with name+PIN auth call
7. **Browser Register** — add PIN field
8. **Browser Login** — add login form alongside Register (for recovery)
9. **Test end-to-end** — register in browser, connect MCP, make moves, verify browser sees them

## Open Questions for Review

1. **PIN length** — 4 digits is frictionless but guessable (10k combos). 6 digits is better (1M). Which do you prefer?
2. **Name uniqueness** — current design allows duplicate names. Should we warn the user if their name is already taken (without blocking registration)?
3. **Old data migration** — the two "kamil" players and "1947" player currently in Railway's SQLite will be wiped when we switch to Supabase. That's fine given the current state — confirm?
4. **MCP_PLAYER_PIN in `.mcp.json`** — this file is gitignored? Check before committing PIN to the repo.
